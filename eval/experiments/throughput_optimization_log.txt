XDN PB WordPress Throughput Optimization Experiment Log
================================================================================
Goal: Beat OpenEBS 3-replica throughput at 1500 rps
  OpenEBS baseline: 1489 rps achieved, 31.9ms avg latency at 1500 rps offered
  XDN PB baseline:  1354 rps achieved, 375ms avg latency at 1500 rps offered

All tests: WordPress 6.5.4 (pinned), editPost XML-RPC, 10s duration, 5s warmup

Iteration 1: Connection pool fix — close stale channels, retry on inactive
--------------------------------------------------------------------------------
Date: 2026-03-22
Hypothesis: At 1500 rps, the screen log showed 1104 "HTTP fwd retry" warnings
  from ClosedChannelException and "Backend channel closed". Apache WordPress
  sends Connection: close, so pooled channels become stale. The retry cycle
  (acquire stale -> fail -> retry) adds latency and wastes throughput.

Fix:
  1. In RequestContext.complete(): if response has Connection: close, close the
     channel instead of returning to pool.
  2. In executeOnce(): if acquired channel is inactive, close it and acquire a
     fresh one instead of throwing IllegalStateException.
  3. Pre-warm pool with 8 connections on first use.

Result: FAILURE — system completely broke. 3.8 rps throughput, 16.3s avg latency.
  7501 requests sent, only 125 succeeded, 3280 got 503, 4096 timed out.
  Root cause: channel.close() without pool.release() caused FixedChannelPool's
  acquired count to never decrement. Pool slots exhausted, no new connections
  could be created.

Note: Retry warnings went to 0 (the close logic worked), but pool exhaustion
  was worse than retries.


Iteration 2: Fix pool slot leak — release before close
--------------------------------------------------------------------------------
Date: 2026-03-22
Hypothesis: FixedChannelPool tracks acquired channels. channel.close() without
  pool.release() keeps the slot occupied forever. Must release() first to
  decrement the acquired count, THEN close() to invalidate the channel.

Fix:
  1. RequestContext.complete(): release() then channel.close() for Connection:close
  2. PipelineContext: releaseChannel() then channel.close()
  3. executeOnce() retry: releaseQuietly() then channel.close() for inactive channels

Result: System functional again. No retries, no pool exhaustion.
  Achieved: 1358 rps, 323ms avg, p50=160ms, p95=956ms
  All 15015 requests succeeded (HTTP 200).
  0 retry warnings, 0 ClosedChannelException.
  6 null responses (down from hundreds), 14 lines with 503.

  Compared to pre-fix baseline (1354 rps, 375ms avg): marginal improvement.
  The connection pool fix eliminated retry overhead but throughput is still
  ~1358 rps — same saturation point. The bottleneck is NOT the connection pool.

Analysis: The connection pool was causing warnings but not the throughput limit.
  At 1500 rps, XDN achieves ~1358 rps regardless of pool behavior. The
  bottleneck must be elsewhere — likely the single capture thread or
  WordPress/Apache processing capacity.

Next: Need to identify the actual throughput bottleneck. Check if WordPress
  Apache is saturated (MaxRequestWorkers) or if the capture thread can't
  keep up at this rate.

Iteration 3: Adaptive capture accumulation window
--------------------------------------------------------------------------------
Date: 2026-03-22
Hypothesis: At high load, the fixed 0.1ms accumulation window results in many
  small proposals (~1 req/proposal). The NVMe fsync throughput and capture
  thread serialization limit proposal rate to ~1350/sec. Adaptive window:
  grow toward 5ms under load (more batching), shrink to 0.1ms when idle
  (low latency).

Fix:
  - CAPTURE_ACCUMULATION_MIN_US = 100 (0.1ms, from PB_CAPTURE_ACCUMULATION_US)
  - CAPTURE_ACCUMULATION_MAX_US = 5000 (5ms, from PB_CAPTURE_ACCUMULATION_MAX_US)
  - After each drain: if doneQueue still has items → double the window
    (capped at MAX); if queue empty and only 1 item drained → reset to MIN.

Result:
  Rate   Achieved   Avg      p50      p95      p99
  1000    991 rps   20.4ms   19.2ms   26.3ms   55.0ms
  1200   1210 rps   23.9ms   21.1ms   38.2ms   45.0ms
  1400   1331 rps  176.8ms   37.4ms  567.8ms  606.9ms
  1600   1364 rps  672.2ms  533.9ms  1656ms   1763ms

  vs Iteration 2b:
  1000    982 rps   19.5ms   19.1ms   23.8ms   26.5ms
  1200   1197 rps   24.2ms   21.2ms   40.4ms   47.2ms
  1400   1322 rps  126.0ms   31.6ms  413.6ms  434.5ms
  1600   1351 rps  657.5ms  487.5ms  1768ms   1851ms

Analysis: Marginal improvement. Throughput at 1600: 1364 vs 1351 (+13 rps).
  Low-load latency preserved (~20ms at 1000 rps). But the saturation ceiling
  barely moved. Adaptive accumulation helped slightly at 1400 (1331 vs 1322)
  but the fundamental bottleneck is elsewhere.

  htop observation: only 2 threads at ~100% CPU during the load test.
  The remaining 62 threads are idle. The bottleneck is likely a single-
  threaded component (Netty EventLoop or capture thread) that can't process
  requests fast enough.

  Need to identify which 2 threads are hot. Must run top -H during load.

  htop identified: 2 ForkJoinPool threads at 99.9% and 94.1% CPU.
  These come from executeXdnBatchOfBatches() line 999:
    CompletableFuture.runAsync(() -> replicableApp.execute(xdnBatch))
  This path is used when HTTP_AR_FRONTEND_BATCH_ENABLED=true — requests
  arrive as XdnHttpRequestBatch and are dispatched to ForkJoinPool.commonPool().

  The ForkJoinPool is a CPU bottleneck because:
  1. Each worker takes 1 batch, delegates to ForkJoinPool, then blocks on join()
  2. The ForkJoinPool threads do the actual HTTP forwarding (I/O-bound) but
     managed blocking + compensation threading causes CPU spin
  3. Only 2 FJP threads are hot despite 32 workers — the workers are blocked
     waiting for FJP threads, creating a serialization point

  Hypothesis for iteration 4: disable HTTP_AR_FRONTEND_BATCH_ENABLED so
  requests arrive as individual XdnHttpRequest (not XdnHttpRequestBatch).
  This routes through executeXdnBatch() which calls execute() directly on the
  PBM worker thread — no ForkJoinPool indirection. The 32 workers each do
  their own HTTP forwarding in parallel.


Iteration 4: Disable HTTP frontend batching
--------------------------------------------------------------------------------
Date: 2026-03-22
Hypothesis: ForkJoinPool.commonPool() is the CPU bottleneck at 99% on 2 threads.
  HTTP batching routes requests through executeXdnBatchOfBatches() which delegates
  to ForkJoinPool, creating unnecessary thread indirection. Disabling batching
  routes through executeXdnBatch() where PBM workers execute() directly.

Fix: Set HTTP_AR_FRONTEND_BATCH_ENABLED=false in gigapaxos config.

Result:

  1000   1008 rps   19.5ms   18.8ms   25.1ms   29.4ms
  1200   1208 rps   22.8ms   20.5ms   35.5ms   43.5ms
  1400   1326 rps  112.2ms   31.9ms  369.4ms  417.3ms
  1600   1385 rps  506.7ms  312.2ms  1420ms   1536ms

  All 200 OK, 0 retries, 0 null responses, 0 ForkJoinPool activity.

  Comparison across iterations at 1600 rps offered:
    Iter 2b (batching ON,  fixed accum):  1351 rps, 657ms avg
    Iter 3  (batching ON,  adaptive):     1364 rps, 672ms avg
    Iter 4  (batching OFF, adaptive):     1385 rps, 507ms avg   ← best so far
    OpenEBS baseline:                     1489 rps,  32ms avg   ← target

  Disabling batching improved throughput by +21 rps (1364→1385) and reduced
  avg latency significantly (672→507ms). The ForkJoinPool indirection was
  adding overhead. But we're still ~104 rps short of OpenEBS.

  At 1400 rps: 1326 achieved (94.7%), 112ms avg. The saturation knee is

  Next: Run with profiler to identify remaining bottleneck threads.

  3. Connection pool: release-before-close for Connection:close servers

  5. Pool pre-warming (8 connections)

Iteration 6: JFR + thread dump analysis — ROOT CAUSE IDENTIFIED
--------------------------------------------------------------------------------
Date: 2026-03-23
Profiling: JFR recording + thread dump captured during 1500 rps load.

Finding: ALL 32 PBM worker threads (pb-batch-wordpress-0 through 31) are
  blocked on the same call stack:

    CompletableFuture.get()
      → ForkJoinPool.managedBlock()
      → XdnHttpForwarderClient.executeOnce() line 334
      → responseFuture.get()  ← BLOCKING inside virtual thread

  PBM workers are virtual threads (Thread.ofVirtual()). Virtual threads on
  JDK 21 use ForkJoinPool as carrier threads. When a virtual thread calls
  CompletableFuture.get(), it triggers managedBlock() which can PIN the
  carrier thread, preventing other virtual threads from running.

  With default ForkJoinPool parallelism (~2 active carrier threads on this
  workload), only 2 workers can make progress concurrently. The remaining
  30 workers are parked waiting for carrier threads.

Root cause of tail latency:
  - 32 workers × 16ms per request = 2000 rps theoretical capacity
  - But only 2 carrier threads → 2 × 62.5 req/s = 125 concurrent capacity
  - Actually higher due to I/O unmounting, but convoy effect creates bursts
  - Requests queue behind each other → p50=10ms but p95=1494ms, avg=286ms

  OpenEBS doesn't have this problem because it has no proxy layer — the Go
  client connects directly to Apache, and each Go goroutine independently
  handles its own HTTP connection.

Fix options for next iteration:
  1. Increase jdk.virtualThreadScheduler.parallelism (default = nCPU)
     → Actually default should be 64 on this machine, so something else
     is limiting it to 2 carrier threads.
  2. Replace CompletableFuture.get() with async callback in executeOnce()
  3. Switch PBM workers from virtual threads to platform threads
     (avoids ForkJoinPool carrier entirely)

  UPDATE: JFR analysis disproved virtual thread pinning hypothesis.
  jdk.VirtualThreadPinned = 0 events. Virtual threads unmount properly.
  The 2 hot ForkJoinPool threads are just the active carrier threads at
  that top -H snapshot — normal behavior for I/O-bound virtual threads.

Iteration 6b: JFR root cause analysis — GigaPaxos internal pipeline
--------------------------------------------------------------------------------
Date: 2026-03-23

JFR hot-methods (top 3):
  7.4%  io.netty.util.Recycler$Stack.scavengeSome()   — Netty object recycling
  4.8%  io.netty.util.Recycler$Stack.push()            — Netty recycling
  4.4%  net.minidev.json.JStylerObj$Escape4Web.escape() — JSON serialization

JFR JavaMonitorWait (source of tail latency):
  RequestBatcher       waits ~100ms on HashMap, notified by pb-capture thread
  AbstractPaxosLogger  waits ~100ms on LinkedList, notified by RequestBatcher
  PaxosPacketBatcher   waits ~100ms on itself, notified by MessageNIOTransport

  These are GigaPaxos internal ConsumerTask threads that poll with 100ms
  timeouts. When proposals arrive just after a consumer starts waiting,
  the proposal waits up to 100ms for the consumer to wake up. This creates
  a multi-stage pipeline stall:

    capture thread → propose() → RequestBatcher (wait 100ms)
      → PaxosLogger (wait 100ms) → PaxosPacketBatcher (wait 100ms)
      → NIO transport → backup replicas → commit callback

  Worst case: 3 × 100ms = 300ms added latency for a single proposal if
  it arrives at the wrong time in each consumer's wait cycle. This matches
  the observed avg=286ms at 1500 rps.

JFR GC:
  Young GC every ~3-4s, 7-15ms pauses (not the main issue).
  Heap oscillates 570MB-1000MB, indicating high allocation rate from
  Netty ByteBuf and JSON serialization.

CORRECTION: The 100ms waits in JFR JavaMonitorWait were IDLE waits (threads
  waiting for new work), not stalls. ConsumerTask uses wait()/notifyAll() with
  no timeout — idle threads wait indefinitely until notified.

  jdk.VirtualThreadPinned = 0 events — no virtual thread pinning.

  Actual root cause of tail latency: BIMODAL DISTRIBUTION from batched PB.
  p50=10ms (most requests complete quickly within a batch cycle) but
  p95=1494ms (requests at the start of a large batch wait for the entire
  batch to be proposed and committed via Paxos).

  The adaptive accumulation window grows to 5ms under load. Combined with
  Paxos commit latency (propose → disk fsync → network ACK from backups
  → commit callback), batch cycles can take 10-100ms+. Requests arriving
  at the start of a cycle wait for the full cycle.

  JFR hot methods: Netty Recycler (12%), JSON serialization (4.4%),
  ForkJoinPool scheduling (4%). No single hot method dominates.

  GC: Young GC every ~3-4s, 7-15ms pauses — minor contributor.

  The high tail latency is a fundamental cost of batched PB with a single
  capture thread. The batching that improves throughput (fewer proposals)
  hurts tail latency (requests wait for their batch's Paxos commit).
  OpenEBS avoids this because its replication is per-write, not batched.


Iteration 7: Disable GigaPaxos RequestBatcher (BATCHING_ENABLED=false)
--------------------------------------------------------------------------------
Date: 2026-03-22
Hypothesis: JFR showed RequestBatcher, PaxosLogger, PaxosPacketBatcher forming
  a multi-stage pipeline with wait/notify. The RequestBatcher adds a ConsumerTask
  hop with a sleep (BATCH_SLEEP_DURATION + agreementLatency * BATCH_OVERHEAD)
  before each proposal reaches the Paxos consensus path. With only 1 Paxos group
  (wordpress), batching provides no benefit — it just adds per-proposal latency.
  Disabling it sends proposals directly from capture thread to handlePaxosPacket().

Fix: Set BATCHING_ENABLED=false in gigapaxos.xdn.wordpress-pb.cloudlab.properties

Test: 30s duration, rates 1400,1500,1600

  This is the irreducible cost of synchronous PB replication.


Iteration 8: Increase PBM workers from 32 to 64
--------------------------------------------------------------------------------
Date: 2026-03-23
Hypothesis: PBM_SAMPLE showed periodic WordPress slowdowns (exec 8ms→24ms)
  causing worker queue buildup. With 32 workers at 24ms exec, capacity drops
  to 1333 rps — below the 1500 rps offered rate. 64 workers at 24ms = 2667 rps
  capacity, staying above 1500 even during slowdowns.

Fix: PB_N_PARALLEL_WORKERS=64 (was 32)

Result (30s at 1500 rps):
  Achieved: 1497 rps, Avg=16.2ms, p50=9.8ms, p90=32.5ms, p95=35.9ms, p99=43.0ms

  vs 32 workers:  Avg=366ms, p50=10.6ms, p90=1346ms, p95=1528ms, p99=1713ms
  vs OpenEBS:     Avg=32ms,  p50=26ms,   p90=N/A,    p95=59ms,   p99=67ms

  XDN PB now BEATS OpenEBS at every latency percentile while matching throughput.

  All optimizations applied:
  1. PB_CAPTURE_ACCUMULATION_US=100 (adaptive, 100µs min → 5ms max)
  2. Connection pool: release-before-close for Connection:close servers
  3. Pre-emptive fresh connection on inactive channel acquire
  4. Pool pre-warming (8 connections)
  5. HTTP_AR_FRONTEND_BATCH_ENABLED=false — eliminated ForkJoinPool bottleneck
  6. BATCHING_ENABLED=false — eliminated RequestBatcher hop
  7. XDN_SKIP_INIT_SYNC=true — skip rsync for faster benchmark setup
  8. XDN_MIGRATE_WAIT_SEC=5 — reduced MySQL migration wait
  9. PB_N_PARALLEL_WORKERS=64 — eliminated queue buildup during WP slowdowns
